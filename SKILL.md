# 多模型团队协作系统

把 AI 模型组织成一个专业团队，像公司一样协作完成项目。

## 团队角色配置

| 角色 | 推荐模型 | 核心能力 | 职责 |
|------|----------|----------|------|
| **产品经理** | Qwen-Plus | 需求分析、逻辑规划、用户体验 | 需求文档、功能设计、优先级排序 |
| **程序员** | GLM-5 | 代码生成、Bug修复、逻辑实现 | 核心开发、功能实现、技术方案 |
| **美术设计师** | Qwen3.5-Plus | 视觉设计、创意表达、UI/UX | 界面设计、配色方案、交互优化 |
| **测试工程师** | DeepSeek-Coder | Bug发现、测试用例、边界测试 | 功能测试、回归测试、问题定位 |
| **代码审查员** | Qwen-Max | 代码审查、性能优化、安全检查 | 代码质量、性能分析、最佳实践 |

## 工作流程

### 阶段一：需求分析（产品经理主导）

```
输入：用户需求
输出：需求文档 + 功能列表 + 优先级

产品经理职责：
1. 分析用户真实需求
2. 拆解功能模块
3. 定义验收标准
4. 排列优先级（P0/P1/P2）
```

### 阶段二：设计与开发（程序员 + 设计师协作）

```
输入：需求文档
输出：代码实现 + UI设计

程序员职责：
1. 技术方案设计
2. 核心代码编写
3. 接口实现

设计师职责：
1. UI/UX设计
2. 配色方案
3. 交互动效
```

### 阶段三：测试验收（测试工程师主导）

```
输入：代码实现
输出：测试报告 + Bug列表

测试工程师职责：
1. 功能测试
2. 边界测试
3. Bug定位与复现
4. 回归测试
```

### 阶段四：代码审查（审查员主导）

```
输入：通过的代码
输出：审查报告 + 优化建议

代码审查员职责：
1. 代码质量检查
2. 性能分析
3. 安全审计
4. 最佳实践建议
```

### 阶段五：交付

```
输出：完整项目 + 文档
```

## 使用方法

### 启动团队协作

告诉用户：
```
我现在以【角色名】身份工作，负责【职责】。
```

### 角色切换

完成一个阶段后，主动切换：
```
【角色A】工作完成，现在切换到【角色B】...
```

### 角色声明模板

```markdown
## 当前角色：[角色名]

**模型能力**：[能力描述]
**当前任务**：[具体任务]

### 工作内容
- [ ] 任务1
- [ ] 任务2
...

### 输出
[具体输出]
```

## 决策树

```
用户提出项目需求
    ↓
是需求分析阶段？ → 产品经理介入
    ↓
是开发阶段？ → 程序员 + 设计师
    ↓
是测试阶段？ → 测试工程师
    ↓
是审查阶段？ → 代码审查员
    ↓
交付
```

## 注意事项

1. **一次一个角色**：每个阶段专注一个角色，避免混乱
2. **明确交接**：阶段切换时明确说明"角色A完成，切换到角色B"
3. **保留上下文**：每个角色的输出是下一个角色的输入
4. **质量把控**：测试和审查阶段不能跳过
